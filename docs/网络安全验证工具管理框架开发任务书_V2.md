# 网络安全验证工具管理框架开发任务书

> 文档版本：V2.0  
> 文档性质：开发任务书 / 产品技术规划 / 接入标准设计草案  
> 项目名称：网络安全验证工具管理框架  
> 核心定位：为各类网络安全验证工具提供统一接入、统一编排、统一执行、统一输出、统一审计和统一协同的管理框架  
> 建设原则：框架优先、标准优先、CLI 优先、插件化、可扩展、可审计、可持续进化  
> 安全边界：仅服务于已授权资产、已授权任务、已授权人员、已授权时间窗口内的安全验证与风险评估

2. 将项目重点从“提供具体工具功能”调整为“管理、接入、对接和使用各种安全验证工具”。

7. 对高风险能力继续采用授权、审批、隔离、审计、限速、靶场化和低影响验证方式进行约束。

## 2. 项目背景

网络安全评估工作中通常会使用大量工具，包括但不限于：

- 子域名探测工具
- DNS 解析工具
- 资产发现工具
- 端口探测工具
- 服务识别工具
- Web 指纹识别工具
- 漏洞扫描工具
- 漏洞验证工具
- 弱口令风险检测工具
- 配置基线检查工具
- 证书检查工具
- 报告生成工具
- 外部暴露面分析工具
- 资产情报查询工具

这些工具生态丰富，但实际使用中存在明显问题：

1. 工具分散，缺少统一管理。
2. 每个工具的参数、输入、输出、日志格式差异较大。
3. 不同工具之间难以自然衔接。
4. 工具执行结果难以标准化沉淀。
5. 工具执行过程缺少统一审计、授权和安全控制。
6. 安全人员需要重复编写脚本进行数据转换和流程串联。
7. 后续新增工具时，容易产生新的适配成本和维护成本。
8. 图形化平台一旦和具体工具强绑定，后续扩展会变得困难。

因此，本项目不应优先建设一个“内置大量扫描能力的平台”，而应建设一个**网络安全验证工具管理框架**。

框架的核心价值是：

> 为各种安全验证工具提供统一接入标准、统一执行入口、统一任务协议、统一输出规范、统一审计机制和统一协同流程，使不同工具可以被低成本接入、编排、执行、复用、替换和持续演进。

## 3. 项目定位

### 3.1 正确定位

本项目定位为：

> 网络安全验证工具管理框架，是一个面向授权安全评估场景的工具接入、任务编排、执行调度、结果归一、审计留痕和协同使用框架。

框架自身不追求替代所有安全工具，而是解决以下问题：

1. 如何统一管理不同类型的安全验证工具。
2. 如何让不同工具以标准方式接入框架。
3. 如何让不同工具的输入、输出、错误、日志和证据结构统一。
4. 如何让工具之间形成可编排的任务链。
5. 如何让工具运行过程可控、可审计、可复现。
6. 如何让工具能力可以持续扩展。
7. 如何先以 CLI 方式快速落地，再演进为 Web/图形化管理界面。

### 3.2 非目标

本框架不以实现具体扫描能力为核心目标。

不应把第一版重点放在：

1. 自研子域名扫描算法。
2. 自研端口扫描引擎。
3. 自研漏洞扫描器。
4. 自研漏洞利用模块。
5. 自研弱口令爆破器。
6. 自研隐蔽探测能力。
7. 自研持久化能力。
8. 自研攻击链自动化能力。

这些能力如需使用，应通过**插件、适配器、外部工具对接、靶场化验证、受控执行策略**接入框架。

### 3.3 高风险能力边界

框架允许管理高风险安全验证工具，但必须遵守以下原则：

1. 框架不内置真实攻击、持久化、隐蔽绕过、自动化入侵链路等不可控能力。
2. 框架可以定义高风险工具的接入标准、审批机制、隔离机制和审计机制。
3. 框架可以支持靶场环境中的受控验证。
4. 框架可以支持低影响、只读型、证明型验证。
5. 框架必须阻止未授权目标、未审批任务和未授权用户执行高风险工具。
6. 所有高风险工具必须声明能力边界、风险等级、执行条件、输入限制、输出格式和审计要求。

## 4. 建设目标

本项目最终目标是建设一套可持续进化的网络安全验证工具管理框架。

### 4.1 核心目标

1. 提供统一的 CLI 使用入口。
2. 提供统一的工具接入标准。
3. 提供统一的插件/适配器规范。
4. 提供统一的任务描述协议。
5. 提供统一的工具输入协议。
6. 提供统一的工具输出协议。
7. 提供统一的错误码和日志规范。
8. 提供统一的证据和结果模型。
9. 提供统一的审计和授权机制。
10. 提供统一的任务编排能力。
11. 提供统一的本地配置、项目配置和运行上下文。
12. 后续提供 Web/API/图形化界面。

### 4.2 阶段目标

第一阶段：实现 CLI 框架和标准体系。

第二阶段：实现插件系统、适配器 SDK 和常见工具接入样例。

第三阶段：实现任务编排、结果归一、报告生成和审计能力。

第四阶段：实现 Web/API 管理界面。

第五阶段：实现工具生态、插件市场、规则库、知识库和 AI 辅助分析。

## 5. 总体建设思路

本框架采用“核心框架 + 接入标准 + 插件适配器 + CLI + API + Web UI”的渐进式架构。

推荐演进路径：

```text
阶段 1：CLI Core
  ↓
阶段 2：Plugin Runtime / Adapter SDK
  ↓
阶段 3：Task Orchestration / Result Normalize
  ↓
阶段 4：Local Service API
  ↓
阶段 5：Web Console
  ↓
阶段 6：Plugin Ecosystem / Team Collaboration
```

设计原则：

1. CLI 先行，避免一开始被 Web 页面拖慢。
2. 标准先行，避免工具接入混乱。
3. 插件优先，避免核心框架和具体工具强耦合。
4. 配置即能力，工具、任务、输出、审计均可通过配置描述。
5. 本地可用，团队可用，平台可用。
6. 单机可运行，后续可扩展到分布式 Worker。
7. 结果标准化，便于后续进入 Web、报告、整改和数据分析流程。

## 7. 核心能力设计

## 7.1 CLI 优先能力

CLI 是本框架的第一产品形态，应优先实现。

### 7.1.1 CLI 目标

1. 不依赖 Web 即可使用核心能力。
2. 便于安全工程师在终端、脚本、CI/CD、自动化流程中使用。
3. 便于快速验证工具接入标准。
4. 便于后续 Web 界面直接复用 CLI/Core 能力。
5. 降低第一阶段开发复杂度。

### 7.1.2 CLI 基础命令设计

```bash
secfw init
secfw config show
secfw tool list
secfw tool info <tool_name>
secfw tool validate <tool_manifest>
secfw tool run <tool_name> --input input.json
secfw task run task.yaml
secfw task plan task.yaml
secfw task status <task_id>
secfw task logs <task_id>
secfw result normalize raw.json --schema finding
secfw result export --format json|md|html
secfw report generate --task <task_id>
secfw audit list
secfw plugin install <plugin_path>
secfw plugin validate <plugin_path>
```

### 7.1.3 CLI 第一版必须支持

1. 初始化工作目录。
2. 加载全局配置。
3. 加载项目配置。
4. 注册工具。
5. 校验工具 Manifest。
6. 执行单个工具。
7. 执行任务文件。
8. 输出标准 JSON 结果。
9. 保存任务日志。
10. 生成基础 Markdown 报告。
11. 查看审计记录。

## 7.3 插件与适配器体系

### 7.3.1 插件和适配器的区别

| 概念 | 说明 |
|---|---|
| 插件 Plugin | 一组可被框架安装、加载、验证和执行的扩展包 |
| 适配器 Adapter | 将某个具体工具转换为框架标准协议的桥接层 |
| 运行器 Runner | 实际执行工具的程序或脚本 |
| 解析器 Parser | 将工具原始输出转换为框架标准输出 |
| Manifest | 描述工具能力、参数、风险、执行方式和输出格式的元数据文件 |

### 7.3.2 插件目录结构

```text
plugins/
  example_tool/
    secfw.tool.yaml
    runner/
      runner.py
    parser/
      parser.py
    schemas/
      input.schema.json
      output.schema.json
    examples/
      input.example.json
      output.example.json
    README.md
```

### 7.3.3 插件生命周期

```text
发现插件
  ↓
读取 Manifest
  ↓
校验 Schema
  ↓
检查依赖
  ↓
安全评估
  ↓
注册工具
  ↓
测试运行
  ↓
启用工具
  ↓
执行任务
  ↓
解析输出
  ↓
版本升级
  ↓
停用/卸载
```

## 8. 统一接入标准设计

统一接入标准是本框架的核心能力，应作为第一阶段重点建设内容。

## 8.1 标准体系总览

框架至少需要定义以下 10 类标准：

| 标准 | 作用 |
|---|---|
| Tool Manifest 标准 | 描述工具身份、能力、风险、执行方式和参数 |
| Capability 标准 | 描述工具具备的能力类型和能力边界 |
| Input Schema 标准 | 约束工具输入参数 |
| Output Schema 标准 | 约束工具标准输出 |
| Task Spec 标准 | 描述任务流程、目标、工具、策略和上下文 |
| Finding 标准 | 描述工具发现的原始结果 |
| Evidence 标准 | 描述结果证据 |
| Error 标准 | 统一错误码、错误类型和处理方式 |
| Audit Event 标准 | 统一审计事件 |
| Policy 标准 | 描述授权、限速、并发、审批和安全策略 |

这些标准应满足：

1. 可版本化。
2. 可向后兼容。
3. 可扩展字段。
4. 可被机器校验。
5. 可被人类阅读。
6. 可被 CLI、API、Web 共同使用。
7. 可支持本地执行、容器执行、HTTP 对接和远程 Worker 执行。

---

## 8.2 Tool Manifest 标准

### 8.2.1 设计目标

Tool Manifest 是框架接入工具的核心描述文件。

它应解决：

1. 这个工具是谁。
2. 这个工具能做什么。
3. 这个工具如何运行。
4. 这个工具需要什么输入。
5. 这个工具输出什么结果。
6. 这个工具风险等级如何。
7. 这个工具是否需要授权和审批。
8. 这个工具如何被审计。
9. 这个工具是否支持限速、超时、并发控制。
10. 这个工具属于哪个能力类型。

### 8.2.2 Manifest 示例

```yaml
apiVersion: secfw.io/v1
kind: Tool
metadata:
  name: example-web-fingerprint
  displayName: 示例 Web 指纹识别工具
  version: 1.0.0
  vendor: internal
  maintainer: security-team
  description: 用于授权 Web 资产的指纹识别

spec:
  type: web_fingerprint
  category: asset_identification
  riskLevel: low

  capabilities:
    - id: web.fingerprint
      name: Web 指纹识别
      impact: read_only
      requiresApproval: false

  runtime:
    mode: command
    command: python runner/runner.py
    timeoutSeconds: 120
    workingDirectory: .
    env:
      - name: PYTHONUNBUFFERED
        value: "1"

  input:
    schema: schemas/input.schema.json
    examples:
      - examples/input.example.json

  output:
    schema: schemas/output.schema.json
    format: jsonl
    parser: parser/parser.py

  controls:
    requiresAuthorization: true
    requiresApproval: false
    supportsRateLimit: true
    supportsConcurrencyLimit: true
    maxConcurrency: 5
    defaultTimeoutSeconds: 120

  audit:
    recordInput: true
    recordOutputSummary: true
    recordRawOutput: false
    evidenceRequired: true

  compatibility:
    frameworkVersion: ">=1.0.0"
    os:
      - linux
      - macos
```

## 8.3 Capability 标准

### 8.3.1 能力分类

工具能力不应随意填写，需要统一枚举和扩展机制。

建议第一版定义以下能力类型：

```text
asset.subdomain
asset.dns_resolve
asset.http_probe
asset.port_probe
asset.service_detect
asset.web_fingerprint
asset.tls_certificate
risk.web_scan
risk.config_check
risk.weak_password_check
risk.vuln_verify
risk.baseline_check
report.generate
data.import
data.export
```

### 8.3.2 能力风险等级

```text
info      信息采集，不产生主动验证
low       低影响验证，一般只读或轻量请求
medium    中等影响验证，可能产生一定业务影响，需要限速
high      高风险验证，必须审批、隔离、审计
critical  仅允许靶场或特殊授权环境
```

### 8.3.3 能力影响类型

```text
read_only       只读
low_impact      低影响
state_check     状态检查
auth_check      授权验证
credential_check 凭据风险检测
active_verify   主动验证
lab_only        仅限靶场
forbidden       禁止在生产环境执行
```

## 8.4 Input Schema 标准

所有工具输入必须用 Schema 约束，避免参数混乱。

### 8.4.1 标准输入结构

```json
{
  "context": {
    "task_id": "task_xxx",
    "run_id": "run_xxx",
    "operator": "user_xxx",
    "workspace": "default",
    "authorization_scope": "scope_xxx"
  },
  "target": {
    "type": "domain",
    "value": "example.com",
    "metadata": {}
  },
  "options": {
    "timeout": 120,
    "rate_limit": 10,
    "concurrency": 3
  },
  "policy": {
    "allow_active_verify": false,
    "allow_high_risk": false,
    "allowed_time_window": "22:00-06:00"
  }
}
```

### 8.4.2 输入要求

1. 所有输入必须携带上下文。
2. 所有输入必须携带授权范围。
3. 所有输入必须明确目标类型。
4. 所有输入必须支持 options。
5. 所有输入必须支持 policy。
6. 高风险工具必须显式声明 allow_high_risk。
7. 未通过授权校验的输入不得执行。

```json
    "summary": "基于响应头和页面特征识别",
    "items": []
  },
  "raw": {
    "retained": false,
    "location": null
  }
}
```

### 8.5.2 输出要求

1. 输出必须是 JSON 或 JSON Lines。
2. 输出必须包含工具名称和版本。
3. 输出必须包含任务 ID 和运行 ID。
4. 输出必须包含目标信息。
5. 输出必须包含结果类型。
6. 输出必须包含置信度。
7. 输出必须支持证据结构。
8. 原始输出是否保留由策略控制。
9. 输出必须可被框架统一解析。
10. 输出结构必须支持后续扩展。

## 8.6 Task Spec 标准

Task Spec 用于描述一个完整任务，支持单工具执行和多工具编排。

### 8.6.1 单工具任务示例

```yaml
apiVersion: secfw.io/v1
kind: Task
metadata:
  name: web-fingerprint-task
  description: 对授权 Web 资产执行指纹识别

spec:
  authorizationScope: scope-prod-web
  targets:
    - type: url
      value: https://example.com

  steps:
    - id: fingerprint
      tool: example-web-fingerprint
      input:
        options:
          timeout: 120
          rate_limit: 5

  policy:
    maxConcurrency: 3
    requireApproval: false
    stopOnError: false
    allowedTimeWindow: "22:00-06:00"
```

### 8.6.2 编排任务示例

```yaml
apiVersion: secfw.io/v1
kind: Task
metadata:
  name: asset-discovery-flow

spec:
  authorizationScope: scope-example
  targets:
    - type: domain
      value: example.com

  steps:
    - id: subdomain
      tool: subdomain-adapter
      outputAs: subdomains

    - id: dns_resolve
      tool: dns-resolve-adapter
      dependsOn:
        - subdomain
      inputFrom:
        - subdomains
      outputAs: resolved_ips

    - id: http_probe
      tool: http-probe-adapter
      dependsOn:
        - dns_resolve
      inputFrom:
        - resolved_ips
      outputAs: web_assets

  policy:
    maxConcurrency: 5
    rateLimit: 10
    stopOnError: false
```

### 8.6.3 Task Spec 要求

1. 必须声明授权范围。
2. 必须声明目标。
3. 必须声明步骤。
4. 必须声明策略。
5. 支持 dependsOn。
6. 支持 inputFrom 和 outputAs。
7. 支持失败处理策略。
8. 支持并发和限速。
9. 支持审批策略。
10. 支持 Dry Run 计划预览。

## 9. 持续进化能力设计

框架的长期竞争力取决于标准是否能持续进化，而不是第一版内置了多少工具。

## 9.1 标准版本化

所有标准必须带版本号：

```text
secfw.io/v1
secfw.io/v1alpha1
secfw.io/v1beta1
secfw.io/v2
```

版本策略：

1. alpha：实验能力。
2. beta：基本稳定，可试点。
3. v1：生产可用。
4. v2：重大变更。

### 向后兼容要求

1. v1 标准字段不得随意删除。
2. 新字段必须尽量使用可选字段。
3. Manifest 需要声明兼容的框架版本。
4. CLI 需要支持标准迁移检查。
5. 提供 `secfw migrate` 命令辅助升级配置。

## 9.2 扩展字段机制

所有核心对象支持 extensions 字段。

示例：

```yaml
extensions:

    businessSystem: 门户系统
  x-risk:
    businessCriticality: high
```

要求：

4. Web/API 可选择性展示扩展字段。

## 10. Rust 作为核心能力开发语言的可行性评估

## 10.1 结论

建议认真评估并优先考虑 Rust 作为框架核心能力的开发语言，尤其适合以下部分：

1. CLI 主程序。
2. 配置解析与校验。
3. 工具注册中心。
4. 任务执行器。
5. 插件调度器。
6. 本地结果归一化。
7. 日志与审计事件处理。
8. 本地数据库访问。
9. 高并发任务调度。
10. 跨平台二进制分发。

但不建议将所有能力都强制使用 Rust 实现。

更合理的技术路线是：

```text
Rust 负责框架核心、CLI、调度、协议、性能敏感和安全敏感部分
Python/Go/Node/Shell 等负责具体工具适配器和生态兼容
Web 前端使用 React/Vue
Web 后端可选 Rust / Go / Python
```

## 10.2 Rust 的优势

| 维度 | 评估 |
|---|---|
| 性能 | 适合高并发任务调度、日志处理、结果归一化 |
| 安全性 | 内存安全特性适合安全工具框架核心 |
| 分发 | 可以编译为单个二进制，适合 CLI 工具 |
| 跨平台 | 适合 macOS、Linux、Windows 多环境运行 |
| 依赖控制 | 相比脚本语言更容易形成稳定分发物 |
| 并发能力 | Tokio 等异步生态适合任务调度 |
| 类型系统 | 适合定义稳定协议、Schema、Manifest 数据结构 |
| 长期维护 | 适合做框架底座和核心引擎 |

## 10.3 Rust 的不足

| 维度 | 风险 |
|---|---|
| 开发效率 | 业务功能开发速度可能慢于 Python/Go |
| 团队门槛 | 团队需要 Rust 工程经验 |
| 安全工具生态 | 现有安全工具更多是 Python/Go/Shell，需要适配 |
| Web 后台效率 | 纯 Rust Web 后台开发成本可能高于 Go/FastAPI |
| 插件生态 | 如果插件也强制 Rust，会降低生态扩展性 |
| 调试成本 | 对初级团队不够友好 |

## 10.4 推荐 Rust 技术栈

如果选择 Rust 作为核心能力开发语言，建议如下：

| 模块 | 推荐 |
|---|---|
| CLI | clap |
| 配置解析 | serde + serde_yaml + serde_json + toml |
| Schema 校验 | jsonschema |
| 异步运行时 | tokio |
| HTTP 客户端 | reqwest |
| Web API | axum / actix-web |
| 数据库 | sqlx |
| 本地数据库 | SQLite |
| 日志 | tracing |
| 错误处理 | anyhow / thiserror |
| 插件隔离 | command process / Docker / WASI 后续评估 |
| 打包分发 | cargo-dist / cross |

## 10.5 推荐技术路线

### 路线 A：Rust Core + CLI + Web API

适合长期做成严肃框架产品。

```text
Rust CLI
Rust Core
Rust Plugin Runtime
Rust Local API Service
React Web Console
PostgreSQL / SQLite
```

优点：

1. 核心统一。
2. 性能好。
3. 分发方便。
4. 长期稳定。
5. CLI 和 Web API 可复用同一核心能力。

缺点：

1. 开发门槛高。
2. 团队需要 Rust 能力。
3. 初期开发速度可能较慢。

### 路线 B：Rust CLI/Core + Python Adapter SDK + React Web

推荐优先采用。

```text
Rust：CLI、核心框架、协议校验、任务调度
Python：适配器 SDK、工具包装、Parser 示例
React：Web 控制台
PostgreSQL/SQLite：数据存储
```

优点：

1. 核心稳。
2. 接入快。
3. 兼容现有安全工具生态。
4. 适合持续进化。
5. 不强迫所有插件使用 Rust。

缺点：

1. 多语言工程需要规范。
2. SDK 和协议边界要设计清楚。

### 路线 C：Python/FastAPI 起步，后续 Rust 重构 Core

不推荐作为长期框架路线，但适合快速原型。

优点：

1. 开发快。
2. 安全工具接入方便。
3. 团队门槛低。

缺点：

1. CLI 分发体验一般。
2. 长期核心稳定性和性能可能不足。
3. 后续重构成本较高。
4. 标准化能力容易被业务代码污染。

## 10.6 Rust 采用建议

建议采用：

> Rust Core + Rust CLI + 多语言 Adapter SDK + 后续 Web Console

第一阶段：

1. 用 Rust 实现 CLI。
2. 用 Rust 实现 Manifest 校验。
3. 用 Rust 实现 Task Spec 解析。
4. 用 Rust 实现工具注册中心。
5. 用 Rust 实现命令行工具适配器。
6. 用 Rust 实现标准输出归一化。
7. 用 SQLite 保存本地任务和审计记录。
8. 提供 Python Adapter 示例，方便接入现有工具。

第二阶段：

1. 增加 Docker Adapter。
2. 增加 HTTP Adapter。
3. 增加任务编排 DAG。
4. 增加报告生成。
5. 增加本地 API Service。

第三阶段：

1. 增加 Web Console。
2. 增加分布式 Worker。
3. 增加插件市场。
4. 增加团队协同能力。

## 16. 验收标准

### 16.1 CLI 验收

3. 能通过 `secfw tool list` 查看工具。
4. 能通过 `secfw tool run` 执行工具。
5. 能通过 `secfw task run` 执行任务。
6. 能保存标准结果。
7. 能保存任务日志。
8. 能生成审计事件。
9. 能导出 Markdown 报告。
10. 能在 macOS/Linux 环境运行。

### 16.2 标准验收

1. Tool Manifest 可机器校验。
2. Input Schema 可机器校验。
3. Output Schema 可机器校验。
4. Task Spec 可机器校验。
5. Finding 标准可表达常见结果。
6. Audit Event 可覆盖关键操作。
7. Policy 可约束任务执行边界。

9. 支持 extensions 扩展字段。
10. 新增工具不需要修改核心代码。

### 16.3 扩展性验收

1. 可以接入新的命令行工具。

## 18. 给开发团队 / Codex 的实现要求

开发时必须遵守以下要求：

1. 项目名称使用“网络安全验证工具管理框架”，不要再使用“网络安全验证工具管理平台”作为核心名称。
2. 框架重点是管理和使用工具，不是实现具体工具功能。
3. 第一优先级是 CLI 和标准协议。
4. 第二优先级是插件系统和适配器 SDK。
5. 第三优先级才是 Web 使用界面。
6. 不要把具体工具逻辑写死在核心框架中。
7. 不要让 Web 页面直接调用工具。
8. 不要让插件绕过 Core 执行。
9. 不要让工具输出绕过标准化流程。
10. 不要默认开放高风险工具。
11. 所有工具必须通过 Manifest 接入。
12. 所有任务必须通过 Task Spec 描述。
13. 所有结果必须转换为 Finding 标准。
14. 所有关键动作必须产生 Audit Event。
15. 所有安全边界必须通过 Policy 约束。
16. 新增工具不得修改核心代码。
17. 新增适配器应通过扩展点实现。
18. CLI、API、Web 必须复用同一套 Core。
19. 代码结构必须清晰区分 Core、CLI、Adapter、Plugin、API、Web。
20. Rust Core 必须保持稳定、简洁、可测试。

## 20. 最小可运行闭环

第一版必须跑通以下闭环：

```text
secfw init
  ↓
安装/放置示例插件
  ↓
secfw plugin validate
  ↓
secfw tool list
  ↓
secfw tool run example-tool --input input.json
  ↓
框架执行工具
  ↓
工具输出原始结果
  ↓
Parser 转换为标准 Finding
  ↓
框架保存结果
  ↓
框架生成 Audit Event
  ↓
secfw report generate
  ↓
输出 Markdown 报告
```

第二个闭环：

```text
编写 task.yaml
  ↓
secfw task plan task.yaml
  ↓
框架校验授权、策略、工具依赖
  ↓
secfw task run task.yaml
  ↓
按步骤执行工具
  ↓
步骤之间传递标准结果
  ↓
结果归一化
  ↓
保存任务日志和审计事件
  ↓
导出结果
```

## 21. 总结

本项目应从“工具平台”升级为“工具管理框架”。

框架的核心不是提供某一个具体安全工具能力，而是提供：

```text
统一接入标准
统一工具描述
统一任务协议
统一执行入口
统一输出结构
统一结果归一
统一策略控制
统一审计留痕
统一插件体系
统一持续进化机制
```

推荐路线是：

```text
Rust Core + CLI First + Plugin/Adapter Standard + Result Normalize + API/Web Later
```

最终形成：

> 一个以标准和框架为核心的网络安全验证工具管理框架，使各类安全验证工具能够被统一接入、统一管理、统一执行、统一编排、统一审计和持续扩展，先服务于 CLI 场景，再自然演进为 Web/图形化协同使用界面。
